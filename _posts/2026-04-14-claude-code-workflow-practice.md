## 핵심 원칙

**작성된 계획을 직접 검토하고 승인하기 전까지 클로드에게 절대 코드를 작성하게 하지 마라.**
(기획과 코딩의 철저한 분리)

## 1단계: 리서치 (Research)

- 코드를 작성하기 전, 클로드에게 관련 코드베이스를 깊이 읽고 이해하게 지시한다.
- 채팅창에서 구두 요약으로 끝내지 말고, 반드시 `research.md` 파일에 상세한 분석 보고서를 작성하게 한다.
- **프롬프트 팁:** "깊이", "매우 상세히", "모든 세부사항"과 같은 단어를 사용하여 클로드가 대충 훑어보는 것을 방지한다.

## 2단계: 계획 (Planning)

- `plan.md`와 같은 별도의 마크다운 파일에 상세 구현 계획을 작성하게 한다.
- 이 문서에는 접근 방식, 실제 변경을 보여주는 코드 스니펫, 수정될 파일 경로, 고려사항 및 트레이드오프가 모두 포함되어야 한다.
- 클로드의 내장 플랜 모드보다, **실제 마크다운 파일로 남겨** 에디터에서 직접 편집하고 영구적인 산출물로 남기는 것이 좋다.

## 3단계: 주석 달기 반복 (The Annotation Cycle)

워크플로우의 핵심 단계.

- 클로드가 작성한 계획 문서를 에디터에서 열고, 직접 인라인 메모(주석)를 달아 피드백을 준다. (잘못된 가정 수정, 제약 조건 추가 등)
- 메모 작성 후 클로드에게 **"모든 메모를 반영하여 문서를 업데이트 하되, 아직 구현하지 마"** 라고 명시적으로 지시한다.
- 클로드가 혼자 판단하여 코드를 짜기 시작하지 않도록 통제권을 쥐고, 계획이 완벽해질 때까지 이 과정을 반복한다.

## 4단계: 구현 (Implementation)

- 계획이 완전히 준비되면 **"전부 구현해라. 완료될 때까지 멈추지 마라"** 와 같은 표준 프롬프트로 기계적인 코딩을 지시한다.
- 창의적인 판단은 이미 계획 단계에서 끝났으므로, 개발자의 역할은 *'설계자'* 에서 *'감독자'* 로 바뀐다.
- 프론트엔드의 시각적 문제 등은 **스크린샷을 첨부**하여 피드백을 주면 효율적이다.
- 만약 구현 방향이 엇나간다면, 억지로 패치하려 하지 말고 `git`으로 통째로 되돌린 후 범위를 좁혀 다시 시도하는 것이 더 좋은 결과를 낳는다.

## 5단계: 하나의 긴 세션 (One Long Session)

- 리서치, 기획, 구현을 여러 세션으로 쪼개지 않고 하나의 긴 세션에서 이어간다.
- 컨텍스트 윈도우가 꽉 차더라도 클로드의 오토 컴팩션이 충분한 맥락을 유지해주며, 미리 만들어둔 영구적인 계획 문서(`.md`)가 있기 때문에 연속성이 유지된다.

## 정리

클로드 코드로 바이브 코딩을 할 때 가장 중요한 건 **"계획과 구현을 분리"** 하는 것. 계획을 마크다운 파일로 남기고 주석 달기 반복을 통해 완벽하게 다듬은 뒤, 구현은 기계적으로 맡기는 방식이다. 이 방식의 핵심은 개발자가 **'설계자'이자 '감독자'** 역할을 유지하는 데 있다.
